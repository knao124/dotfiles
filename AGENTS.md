# 共通

- 回答はすべて日本語で行うこと
- コード変更を伴う作業では、編集前に必ず `git-worktree-start` skill を使い、`origin/<base>` への同期、専用 worktree の作成、`codex/` プレフィックス付き branch の作成をこの順で実行すること
- 以後の編集・テスト・コミット・PR 作業は、その worktree 上で継続すること

## 開発判断

- ユーザーの依頼内容をそのまま実装する前に、一般的な開発判断としてより妥当な対案があるかを必ず検討する。
- 次の観点で、依頼案より明確に良い案がある場合は、実装前に対案を提示する。
  - 保守性: 既存実装の再利用で済むか、新しい分岐や重複実装を増やさないか
  - 影響範囲: 差分が小さく、既存利用者や既存運用を壊しにくいか
  - 一貫性: 既存の設計、命名、データ配置、運用フローに沿っているか
  - 運用性: 一時対応ではなく、今後も同じ手順で回しやすいか
  - 検証容易性: テストや動作確認、切り戻しがしやすいか
  - 複雑性: 新しい概念や専用スクリプトを増やさずに解けるか
- 特に次の状況では、対案提示を必須とする。
  - 既存コードや既存スクリプトの複製に近い実装になる
  - データ配置や入出力の整形だけで既存フローを再利用できる
  - 専用の新機能を足すより、既存機能への小さな拡張で済む
  - 依頼案だと将来の運用負荷や理解コストが増える
- 対案を提示するときは、短く以下を示す。
  - 依頼案
  - 対案
  - 推奨理由
  - 影響範囲
- 明確により良い案がある場合は、対案を提示してユーザーの意思を確認するまで実装を始めない。
- ただし、ユーザーが `提案不要` `指示通り実装` `まず手を動かして` など明示した場合は、その意図を優先する。
