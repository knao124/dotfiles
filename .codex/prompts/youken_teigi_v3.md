# 指示

- user と質疑を繰り返し、以下の規約に従った要件定義書を書け
- 第一原理思考（First Principles Thinking）を意識して、最も根本的な問題を捉えること
- ドキュメントをアウトプットする際、一切の曖昧さは残してはならない
- そのために足りない情報は user と質疑を行うことで解消せよ
- 質疑の際、モバイル端末からの入力を容易にするため、質問は一覧化し、回答は選択肢で提示して入力しやすくせよ

# 要件定義の規約

開発において、ある機能について「仕様をまとめる」とは、以下 3 点を記載することを指す。

- ① 概要
- ② 要件（Feature 形式）
- ③ UI 概要（外部仕様を含む）

概要で機能の全体像（What）を、要件でシステムが提供すべき機能と受入基準（What）を、

UI 概要でユーザー視点の操作・動作（How）を明確化する。

---

## 1. 概要

概要とは、機能の全体像を簡潔に説明する文章のこと。

機能の存在理由と提供する価値を伝えるために、以下のサブセクション構成で記述する。

第一原理思考（First Principles Thinking）を意識して、最も根本的な問題を捉えること。

### 1.1 目的

機能の目的とは、その機能が解決する根本的な問題と、システムが提供する基本的な動作を記述するもの。

- 誰が（ユーザー役割）、何を（対象データ・処理）、どのように（基本的な処理フロー）行うかを簡潔に記述する
- 機能の存在理由を明確にする
- 必要に応じて、ユーザーが得られる具体的なメリット（効率化、手作業削減、精度向上、状況把握など）を箇条書きで記述してもよい

### 1.2 主要な機能の概要

主要な機能の概要とは、機能を構成する主要な機能群を簡潔に列挙するもの。

- 機能を構成する主要な機能を箇条書きで列挙する
- 詳細は要件セクションで記述するため、ここでは概要のみを記載する

**記述例：**

```markdown
# 要件定義書

## 1. 概要

### 1.1 目的

ユーザーがドキュメントを PDF 化し、システムにアップロードしてデータ化する。複数 PDF を一括処理し、1PDF 内の複数ドキュメントも自動分割する。

**提供する価値：**

- 複数のドキュメントを効率的に一括処理できる
- 自動分割により手作業を削減できる
- 分割結果を確認・修正することで、データ抽出の精度を向上できる
- アップロード履歴を確認することで、処理状況を把握できる

### 1.2 主要な機能の概要

- ドキュメント PDF アップロード機能
- PDF 自動分割・確認・修正機能
- アップロード履歴確認機能
```

---

## 2. 要件

要件とは、システムが提供すべき機能（What）を明確に記述するもの。

各機能を**機能詳細**として記述し、BDD（Behavior Driven Development）の考え方を取り入れる。

第一原理思考（First Principles Thinking）を意識して、最も根本的な問題を捉えること。

### 記載ルール

- 各機能を**機能詳細**として記述する。
  - 機能詳細名は機能を表す名詞句とする。
  - **4 要素（できること／前提・権限・対象範囲／動作ルール／例外・安全）を要約した名詞句とする。**
  - 命名パターン例：
    - 【対象】【行為】（例：ドキュメント一覧表示）
    - 【対象】【行為】【目的】（例：ドキュメント編集機能）
    - 【主体向け】【対象】【行為】（例：本部向け 勤務データ CSV 一括登録）
    - 【主体向け】【対象】【行為】【方式】（例：本部向け 勤務データ CSV 一括登録機能）
    - 【対象】【行為】【対象範囲】（例：勤務データ 一括登録機能（当月））
    - 【対象】【方式】【目的】（例：勤務データ CSV 一括登録機能（初期投入））
  - 目安：日本語 8〜28 文字、略語濫用・動詞文・曖昧語（最適・高速など）は避ける。
- 各機能詳細には以下の構成で記述する。
  1. **機能詳細説明**：ユーザーストーリー形式で記述
     - 「[ユーザー役割]として、[行動]することで、[目的]できるようにする。」
     - または「システムとして、[動作]することで、[目的]できるようにする。」（システム主体の場合）
     - 例：「ユーザーとして、Web 画面からドキュメント PDF をアップロードすることで、処理の対象として登録できるようにする。」
  2. **条件**：機能が満たすべき条件を箇条書きで記述
     - 各条件には「条件: [条件名]」という見出しを付ける
     - 条件の詳細を箇条書きで記述する
     - **条件を記述する際は、4 分法の観点を意識する：**
       - **できること**（ユースケース：誰が何をするか）
       - **前提・権限・対象範囲**（誰が／何に対して可能か、スコープ）
       - **動作ルール**（正常：入力・保存・参照・表示の基準、整合性、トランザクション）
       - **例外・エラー・安全**（異常時の扱い、メッセージ、マスキング、ログ／監査、保持期間）
  3. **例**：条件を具体化した表形式の例
     - 表形式で複数のケースを列挙し、網羅性を確保する
     - 正常系・異常系・境界値を含める
     - 読む人が直感的に理解できる表現を使い、専門用語は避ける。

### 漏れ防止チェック（観点）

- 機能詳細説明：ユーザー役割（またはシステム）、行動、目的が明確か
- 条件：主要なユースケースが列挙されているか
- 条件：条件が明確で曖昧さがないか
- 例：正常系・異常系・境界値が含まれているか
- 表形式：複数のケースが網羅的に記載されているか
- **できること**：主要ユースケースが列挙されているか／非同期・バッチ含むか
- **前提・権限・対象範囲**：操作可能者、対象データ範囲、組織・期間・属性の制限
- **動作ルール（正常）**：入力方法・フォーマット、保存条件、上書き・バージョン、参照・照合、表示ルール
- **例外・エラー・安全**：バリデーション、再試行、ロールバック、部分成功、マスキング、ログ・監査、保管期間

**記述例：**

```markdown
## 2. 要件

### 2.1 機能詳細: ドキュメント PDF アップロード機能

ユーザーとして、Web 画面からドキュメント PDF をアップロードすることで、処理の対象として登録できるようにする。

#### 2.1.1 条件: ユーザーが PDF ファイルをアップロードできる

- ファイル形式は PDF のみ許可する
- ファイルサイズは 500MB まで許可する
- ページ数は 1,000 ページまで許可する
- 認証済みのユーザーのみアップロード可能とする
- アップロード時には重複判定を行わない（重複判定は分割後に行う）
- 上記の条件を満たさない場合はエラーを表示する

例

| ユーザー役割   | ファイル形式 | ファイルサイズ | アップロード可否           |
| -------------- | ------------ | -------------- | -------------------------- |
| ユーザー       | PDF          | 100MB          | ○ アップロード可能         |
| ユーザー       | PDF          | 500MB          | ○ アップロード可能         |
| ユーザー       | PDF          | 501MB          | × エラー表示（上限超過）   |
| ユーザー       | PDF          | 1,000 頁       | ○ アップロード可能         |
| ユーザー       | PDF          | 1,001 頁       | × エラー表示（上限超過）   |
| ユーザー       | JPEG         | 10MB           | × エラー表示（形式不正）   |
| ユーザー       | PNG          | 10MB           | × エラー表示（形式不正）   |
| 未認証ユーザー | PDF          | 100MB          | × エラー表示（認証エラー） |

#### 2.1.2 条件: アップロード進捗・履歴の表示と確認

- アップロード開始時は進捗バー 0% を表示する
- アップロード中は進捗バーとパーセンテージをリアルタイムで更新する
- アップロード完了時は進捗バー 100% と成功メッセージを表示する
- アップロード失敗時はエラーメッセージを表示する
- アップロード進捗（進捗バー、パーセンテージ）を画面上で確認できる
- アップロード履歴（アップロード日時、ファイル名、処理状態）を画面上で確認できる
- 履歴一覧から詳細画面へ遷移できる

例

| 項目               | 表示内容                               | 表示可否 |
| ------------------ | -------------------------------------- | -------- |
| アップロード進捗   | 進捗バー、パーセンテージ               | ○        |
| アップロード履歴   | アップロード日時、ファイル名、処理状態 | ○        |
| 履歴からの詳細確認 | 履歴一覧から詳細画面へ遷移             | ○        |

### 2.2 機能詳細: アップロード履歴確認機能

ユーザーとして、アップロードした PDF の履歴と処理状況を確認することで、過去の処理状況を把握できるようにする。

#### 2.2.1 条件: ユーザーが自分のアップロード履歴を確認できる

- ユーザーは自分がアップロードしたファイルのみ履歴として表示される
- 他のユーザーがアップロードしたファイルは表示されない

例

| ユーザー   | アップロードファイル                  | 表示可否       |
| ---------- | ------------------------------------- | -------------- |
| ユーザー A | ユーザー A がアップロードしたファイル | ○ 表示される   |
| ユーザー A | ユーザー B がアップロードしたファイル | × 表示されない |
| ユーザー B | ユーザー B がアップロードしたファイル | ○ 表示される   |

#### 2.2.2 条件: アップロード履歴に必要な情報を表示する

- アップロード日時（YYYY-MM-DD HH:mm 形式）を表示する
- アップロードした PDF ファイル名を表示する
- ファイルサイズ（MB 単位）を表示する
- 総ページ数を表示する
- 分割されたドキュメント数を表示する
- 処理状態（アップロード中/分割中/確認待ち/確定済み/エラー）を表示する
- エラー発生時はエラーメッセージを表示する

例

| 表示項目         | 表示内容                                       | 表示可否          |
| ---------------- | ---------------------------------------------- | ----------------- |
| アップロード日時 | YYYY-MM-DD HH:mm                               | ○                 |
| ファイル名       | アップロードした PDF ファイル名                | ○                 |
| ファイルサイズ   | MB 単位                                        | ○                 |
| ページ数         | 総ページ数                                     | ○                 |
| 分割結果         | 分割されたドキュメント数                       | ○                 |
| 処理状態         | アップロード中/分割中/確認待ち/確定済み/エラー | ○                 |
| エラー情報       | エラー発生時のエラーメッセージ                 | ○（エラー時のみ） |
```

## 3. UI 概要（外部仕様を含む）

UI 概要は、要件で定義したシステムの機能（What）を、

**ユーザーがどう操作し、どのような応答を確認するか（How）**という視点で記述する。

外部仕様の内容（操作・応答・完了条件）をこの章に統合して記載する。

UI を持たない機能の場合、エンドポイントや詳細シーケンスは技術設計書で扱い、本書では方針レベル（認可・スコープ、リトライ／冪等性、異常系・監査）のみを記載する。

### 記載ルール

UI 概要は、以下の**3 つの小見出し構成**で必ず記述する。

要件セクションで定義した各機能詳細に対応する UI 概要を記述する。

### 3.1 画面構成

- 画面を構成する領域と、その内容を**表形式（2 列）**で記載する。
- 複数の画面がある場合は、各画面ごとに小見出しを分けて記述する。
- 要件セクションの各機能詳細に対応する画面を記述する。

### 3.2 操作フロー

- ユーザーの操作手順と、システムの応答を順番に箇条書きする。
- 複数の操作フローがある場合は、各操作ごとに小見出しを分けて記述する。
- 要件セクションの各機能詳細に対応する操作フローを記述する。

### 3.3 表示ルール

- 画面上の色、メッセージ、状態表示、エラーの出し方などを記載する。
- 複数の画面がある場合は、各画面ごとに小見出しを分けて記述する。
- 表示ルール配下に「エラー表示」を小見出しとして設けることを推奨する（例：3.3.4 エラー表示）。
- 要件セクションの各条件で定義された表示要件に対応する表示ルールを記述する。

- UI を持たない機能は、実装詳細（エンドポイント、シーケンス等）は技術設計書で扱い、本書では方針レベル（認可・スコープ、リトライ／冪等性、異常系・監査）のみ記載する。

**記述例：**

```markdown
## 3. UI 概要（外部仕様を含む）

### 3.1 画面構成

#### 3.1.1 ドキュメント一覧画面

| 領域/要素  | 内容                                                 |
| ---------- | ---------------------------------------------------- |
| ヘッダー   | ページタイトル「ドキュメント一覧」                   |
| テーブル   | ID、書類の種類、薬局名、送信者、申請日、作成日を表示 |
| ソート機能 | 各カラムヘッダーをクリックでソート可能               |
| ページング | ページ下部にページ番号と前後ボタン                   |
| 行クリック | 行をクリックで詳細ページへ遷移                       |

#### 3.1.2 ドキュメント詳細画面

| 領域/要素          | 内容                                                 |
| ------------------ | ---------------------------------------------------- |
| ヘッダー           | ページタイトル「ドキュメント詳細」                   |
| ファイルプレビュー | PDF は iframe、画像は img タグで表示                 |
| 編集フォーム       | 書類の種類、薬局、申請日、許可番号、有効期限の入力欄 |
| 保存ボタン         | フォーム内容を保存                                   |
| 削除ボタン         | ドキュメントを削除                                   |
| 戻るボタン         | 一覧ページへ戻る                                     |

### 3.2 操作フロー

#### 3.2.1 ドキュメント一覧表示

1. ユーザーがドキュメント一覧ページにアクセス
2. システムはユーザーに関連するドキュメントを表示
3. ユーザーはソート機能で並び順を変更可能
4. ユーザーはページングで他のページを閲覧可能
5. ユーザーは行をクリックで詳細ページへ遷移

#### 3.2.2 ドキュメント詳細表示

1. ユーザーが一覧の行をクリック
2. システムは詳細ページへ遷移（`/documents/{id}`）
3. システムは関連ファイルのプレビューを表示
   - PDF ファイル：iframe で表示
   - 画像ファイル（JPEG、PNG、WEBP）：img タグで表示

#### 3.2.3 ドキュメント編集

1. ユーザーが編集フォームの各フィールドを編集
2. 書類種別を変更すると、動的にフィールドの表示・非表示が制御される
3. ユーザーが保存ボタンを押下
4. システムはバリデーションを実行
5. バリデーション通過時：変更を保存し、成功メッセージを表示
6. バリデーションエラー時：フィールドごとにエラーメッセージを表示し、保存処理を行わない

#### 3.2.4 ドキュメント削除

1. ユーザーが削除ボタンを押下
2. システムは確認ダイアログを表示（「このドキュメントを削除しますか？」「この操作は取り消せません」）
3. ユーザーが「削除」を選択：ドキュメントと関連ファイルを削除
4. ユーザーが「キャンセル」を選択：削除処理を中止
5. 削除成功時：ドキュメント一覧ページへ遷移し、成功メッセージを表示
6. 削除失敗時：エラーメッセージを表示し、詳細ページに留まる

### 3.3 表示ルール

#### 3.3.1 ドキュメント一覧

- 一覧の表示ルール（件数・ソート項目・順序・未設定時の表示など）は受入基準の各 Rule に準拠する
- UI は当該 Rule を逸脱しない

#### 3.3.2 ドキュメント詳細

- PDF ファイルは iframe 要素でプレビュー表示
- 画像ファイル（JPEG、PNG、WEBP）は img 要素でプレビュー表示

#### 3.3.3 編集フォーム

- 書類種別が type1 の場合：申請日フィールドを表示
- 書類種別が type2 または type4 の場合：許可番号フィールドを表示
- 書類種別が type2、type3、または type4 の場合：有効期限開始日フィールドを表示
- 書類種別が type2 または type3 の場合：有効期限終了日フィールドを表示
- 書類種別の変更に応じて、フィールドの表示・非表示を動的に制御

#### 3.3.4 エラー表示

- バリデーションエラー：フィールドごとにエラーメッセージを表示
- 削除失敗：エラーメッセージを表示（「削除に失敗しました」「権限がありません」など）
- ネットワークエラー：エラーメッセージを表示し、再試行導線を提示
```

### 付録 A: 境界の宣言

- 本書は要件定義の範囲に留める。
- 通信方式、API 設計、バリデーション詳細、詳細 UI デザイン、性能要件などは**詳細設計書で扱う**。
- データ項目（データ型、必須項目など）は、必要に応じて要件セクションの例や UI 概要の表示ルールに含める。
- 処理手順や業務フローは、要件定義書の本文に含める場合があるが、規約としては要件セクションの機能詳細説明に統合することを推奨する。
